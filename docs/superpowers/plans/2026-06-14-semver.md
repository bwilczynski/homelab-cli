# SemVer Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the CalVer tagging scheme with fully-automated SemVer via semantic-release, so that conventional commit prefixes drive version bumps on every push to main.

**Architecture:** `semantic-release` (via `cycjimmy/semantic-release-action@v6`) runs before GoReleaser in CI, analyzes commits since the last tag, pushes a SemVer tag, and gates GoReleaser on `new_release_published == 'true'`. semantic-release owns tagging; GoReleaser owns binary builds, the GitHub release, and the Homebrew tap.

**Tech Stack:** GitHub Actions, cycjimmy/semantic-release-action@v6 (Node.js 24, bundles semantic-release), goreleaser/goreleaser-action@v6

### Task 1: Add `.releaserc.json`

**Files:**
- Create: `.releaserc.json`

- [ ] **Step 1: Create `.releaserc.json`**

```json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator"
  ]
}
```

Both plugins are bundled with semantic-release — no `package.json` or npm install needed. `@semantic-release/github` is intentionally excluded; GoReleaser owns the GitHub release.

- [ ] **Step 2: Commit**

```bash
git add .releaserc.json
git commit -m "chore: add semantic-release config"
```

### Task 2: Update `release.yml`

**Files:**
- Modify: `.github/workflows/release.yml`

- [ ] **Step 1: Replace the workflow file with the updated content**

Replace the entire contents of `.github/workflows/release.yml` with:

```yaml
name: Release

on:
  push:
    branches:
      - main

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive

      - uses: actions/setup-node@v4
        with:
          node-version: 'lts/*'

      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod

      - name: Generate API client code
        run: make generate

      - name: Lint
        run: go vet ./...

      - name: Test
        run: go test ./...

  release:
    runs-on: ubuntu-latest
    needs: ci
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          submodules: recursive
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: 'lts/*'

      - uses: actions/setup-go@v5
        with:
          go-version-file: go.mod

      - name: Generate API client code
        run: make generate

      - name: Semantic Release
        uses: cycjimmy/semantic-release-action@v6
        id: semantic
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

      - uses: goreleaser/goreleaser-action@v6
        if: steps.semantic.outputs.new_release_published == 'true'
        with:
          version: latest
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          TAP_GITHUB_TOKEN: ${{ secrets.TAP_GITHUB_TOKEN }}
```

Key changes from the original:
- `node-version: '20'` → `node-version: 'lts/*'` (both jobs)
- `Compute and push tag` step removed
- `Semantic Release` step added (before GoReleaser)
- GoReleaser gains `if: steps.semantic.outputs.new_release_published == 'true'`
- GoReleaser args: `release --clean --skip=validate` → `release --clean`

- [ ] **Step 2: Commit**

```bash
git add .github/workflows/release.yml
git commit -m "chore: switch to semver via semantic-release"
```

### Task 3: Push and verify

- [ ] **Step 1: Push the branch and open a PR to main**

```bash
git push origin HEAD
```

Open a PR to `main`. When it merges, the release workflow will run.

- [ ] **Step 2: Verify the CI job passes**

In GitHub Actions, confirm the `ci` job (generate → lint → test) completes green.

- [ ] **Step 3: Verify semantic-release behaviour**

In the `release` job logs, the `Semantic Release` step should either:
- Print `Created tag v1.0.0` (if any `fix:` or `feat:` commits are present since no prior SemVer tag) and set `new_release_published=true`, OR
- Print `There are no relevant changes, so no new version is released.` and set `new_release_published=false`

If commits on this branch include only `chore:` prefixes, no release will be cut. To force `v1.0.0` on first merge, ensure at least one `feat:` or `fix:` commit is present, or push a throwaway `fix:` commit.

- [ ] **Step 4: Verify GoReleaser runs only when expected**

If `new_release_published=true`, confirm GoReleaser completes: check that a GitHub release with binary assets appears and that the Homebrew tap was updated. If `new_release_published=false`, GoReleaser step should be skipped (shown as greyed out in the Actions UI).

- [ ] **Step 5: Verify binary version**

Download the release binary and run:

```bash
./hlctl --version
```

Expected output: `hlctl version v1.0.0` (or whatever version was computed). If it prints `dev`, the `-X main.version={{.Version}}` ldflags injection in `.goreleaser.yaml` was not applied — check the GoReleaser build logs.
